# ADR 001: GROUP BY Over Window Functions for Large-Scale Aggregations
## Status

Accepted

## Decision

Use GROUP BY aggregations combined with joins instead of relying heavily on window functions when generating customer-level fraud indicators.

## Context

Several fraud indicators require aggregated customer information, such as:

- Shared phone detection
- Shared email detection
- Historical customer activity
- Address reuse analysis

A naïve implementation using window functions can generate large intermediate datasets because aggregate values are repeated across every matching row.

## Decision Rationale

GROUP BY produces a compact aggregated dataset that can later be joined back into the scoring pipeline.

Benefits:

- Reduced execution time on large datasets
- Lower memory consumption
- Simpler query plans
- Easier troubleshooting

## Consequences

### Positive

- Better scalability
- Clear separation between aggregation and scoring logic
- Easier maintenance

### Negative

- Additional joins required
- Slightly more verbose SQL

## Alternatives Considered

### Window Functions

Pros:

- Compact syntax

Cons:

- Larger intermediate datasets
- Less efficient for repeated aggregations

### Precomputed Aggregate Tables

Pros:

- Fast query execution

Cons:

- Additional maintenance complexity

## Outcome

GROUP BY became the preferred aggregation strategy throughout the fraud-detection pipeline.
